# 提交规范 Contributing Guide

本文档规定本项目的 Git 提交规范，请所有组员在推送代码前阅读并遵守。

---

## 一、提交信息格式

每条提交信息由两部分组成：**类型标签** + **简短描述**。

```
[类型] 简短描述（不超过 50 字）

（可选）正文：说明做了什么、为什么这样做，每行不超过 72 字。
```

### 类型标签

| 标签 | 适用场景 | 主要负责人 |
|------|----------|------------|
| `[论文]` | 修改 `paper/` 下的 `.tex`、`.bib` 文件 | 论文手 |
| `[代码]` | 修改 `code/` 下的 Python 脚本、Notebook | 代码手 |
| `[建模]` | 修改 `model/` 下的假设、推导文档 | 建模手 |
| `[数据]` | 添加或更新 `code/data/` 下的数据文件 | 代码手 |
| `[图表]` | 添加或更新 `paper/figures/`、`code/results/figures/` | 代码手／论文手 |
| `[结果]` | 添加或更新 `code/results/tables/` 下的结果文件 | 代码手 |
| `[文档]` | 修改 `assets/`、`README.md`、`CONTRIBUTING.md` 等说明文件 | 全员 |
| `[修复]` | 修复代码 bug、LaTeX 编译错误、数据处理错误 | 全员 |
| `[结构]` | 目录调整、文件重命名、`.gitignore` 修改 | 全员 |

---

## 二、提交示例

**好的提交信息：**

```
[论文] 完成问题一建模背景与假设初稿

补充了基本假设，引用了附件表1的统计结论。
```

```
[代码] 问题二：实现基于贪心的分配算法
```

```
[数据] 添加附件表原始数据至 code/data/raw/
```

```
[修复] 修正问题三 ROI 计算的单位错误
```

**不好的提交信息（避免）：**

```
update           ← 过于模糊，不知道改了什么
修改了一些东西    ← 没有类型标签，描述不清
fix bug          ← 哪个 bug？在哪里？
wip              ← 不完整的工作不应提交到 main
```

## 三、提交前检查清单

- [ ] 代码可以正常运行，不存在语法错误
- [ ] LaTeX 文件可以正常编译（`cd paper && latexmk -xelatex -outdir=build main.tex`）
- [ ] 没有提交 `paper/build/` 下的编译产物
- [ ] 没有提交 `.DS_Store`、`__pycache__/`、`.ipynb_checkpoints/` 等无关文件
- [ ] 数据文件单个不超过 10 MB（超出请使用 Git LFS 或联系论文手处理）
- [ ] 提交信息包含正确的类型标签

---

## 四、分支与推送规范

### 主分支保护

- `main` 分支为唯一主分支，始终保持**可编译、可运行**的状态
- 不要在 `main` 上直接进行大规模修改，完成后再提交

### 每次工作前先拉取

```bash
git pull origin main
```

### 推送前解决冲突

```bash
git add <冲突文件>
git commit -m "[修复] 解决与 main 的合并冲突"
git push origin main
```

## 五、各角色文件权限

遵守以下分工，可大幅减少冲突：

| 目录 | 负责人 | 其他人 |
|------|--------|--------|
| `paper/main.tex` | 论文手 | 修改前请通知论文手 |
| `paper/sections/` | 论文手 | 各自负责自己的章节 |
| `paper/references.bib` | 论文手 | 需要添加文献时告知论文手 |
| `code/src/` | 代码手 | 建模手可提修改意见 |
| `model/` | 建模手 | 其他人只读参考 |
| `code/data/raw/` | **只读** | 任何人不得修改原始数据 |

## 六、紧急情况处理

### 误提交了不该提交的文件

```bash
git rm --cached <文件路径>
git commit -m "[结构] 移除误提交的文件"
```

### 想撤回上一次提交（未推送）

```bash
git reset --soft HEAD~1
```

### 拉取时遇到冲突

```bash
git status
# 手动编辑文件，解决 <<<< ==== >>>> 标记的冲突
git add <文件>
git commit -m "[修复] 解决合并冲突"
```
